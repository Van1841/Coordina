## Data flow: a message becomes a recommendation

1. **Observe** вҖ” a Slack message lands in any connected workspace. `registerHandlers.js` receives it and hands it to `incidentService.ingestMessage`.
2. **Understand** вҖ” `llm/classifier.js` asks Gemini (falling back to Groq, falling back to keyword heuristics) to classify the message into `{kind, category, peopleAffected, summary}`.
3. **Deduplicate** вҖ” `rts.search()` checks the live semantic index for a similar open incident. A strong match updates the existing incident (e.g. "we now need 200 blankets" restates rather than duplicates); otherwise a new incident row is created.
4. **Score (deterministic)** вҖ” `scoring/priorityEngine.js` pulls live data from every MCP server (inventory shortages, shelter occupancy, volunteer availability, travel time) and computes a weighted 0вҖ“100 score. This step never calls an LLM.
5. **Publish** вҖ” the incident emits an RTS signal. `agents/proactiveLoop.js`, subscribed to the event bus, picks it up immediately (no polling).
6. **Plan** вҖ” `recommendationService.decideAction` deterministically picks an action (`dispatch` / `reallocate` / `escalate` / `monitor`) using MCP data, then asks the LLM router only to explain that decision in prose.
7. **Coordinate** вҖ” if the action targets another organization, Coordina DMs that workspace directly via its own Bolt app instance вҖ” proactively, without a human asking.
8. **Escalate** вҖ” a periodic sweep checks for incidents open past the escalation threshold and posts an escalation card, buttons and all, back into the originating workspace.
9. **Observe (dashboard)** вҖ” every step above also emits into the same RTS event bus, which the Express/Socket.IO server rebroadcasts to the React dashboard's live activity stream вҖ” the exact same event-driven mechanism, reused for humans watching rather than the agent itself.
